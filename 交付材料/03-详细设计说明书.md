## 1. 后端分层设计

| 层次 | 代表类/目录 | 设计说明 |
| --- | --- | --- |
| Controller | `AuthController`、`RoomController`、`ReservationController` 等 | 对外暴露 REST API，调用 Service，并返回 `ApiResponse` |
| Service | `RoomService`、`CheckInService`、`BillingService` 等 | 定义业务能力和事务边界 |
| ServiceImpl | `RoomServiceImpl`、`CheckInServiceImpl` 等 | 编排 Mapper、校验业务状态、更新数据库 |
| Mapper | `RoomMapper`、`BillMapper`、`ReportMapper` 等 | 使用 MyBatis 操作表数据 |
| DTO | `dto.request`、`dto.response` | 隔离接口字段和数据库实体 |

## 2. 状态设计

| 业务对象 | 状态 | 说明 |
| --- | --- | --- |
| 房间 | `vacant` | 空净，可预订或入住 |
| 房间 | `occupied` | 已入住 |
| 房间 | `reserved` | 已预订 |
| 房间 | `dirty` | 退房后待清洁 |
| 房间 | `cleaning` | 清洁处理中 |
| 房间 | `maintenance` | 维修，不可入住 |
| 清洁任务 | `pending/doing/done` | 待处理、进行中、已完成 |
| 预订 | `pending/cancelled/checked_in` | 待入住、已取消、已入住 |

## 3. 关键设计点

- 统一响应：所有接口返回 `code/message/data`，便于前端统一处理。
- 鉴权：除登录和测试账号接口外，其余接口通过 `Authorization: Bearer <token>` 访问。
- 角色校验：需要限制权限的接口使用角色注解或拦截逻辑。
- 金额计算：账单金额使用 `BigDecimal`，避免浮点误差。
- ID 生成：业务 ID 使用前缀加日期/序列，便于测试追踪。
- WebSocket：房态变更后可通过 `/ws/room-status` 推送，当前测试未验证客户端订阅。

## 5. 异常处理设计

| 类型 | 处理方式 |
| --- | --- |
| 参数错误 | 返回 `400` 业务码和参数提示 |
| 未登录/Token 失效 | 返回 `401` 业务码 |
| 无权限 | 返回 `403` 业务码 |
| 业务冲突 | 返回 `409` 业务码，例如房间不可入住 |
| 服务异常 | 返回 `500` 业务码，建议后续补充稳定非空错误消息 |

## 6. 可改进项

- 数据库凭据和 token secret 应从环境变量读取。
- 前端 API 地址应支持构建环境或运行环境配置。
- WebSocket 客户端订阅和浏览器 E2E 测试需要补测。
